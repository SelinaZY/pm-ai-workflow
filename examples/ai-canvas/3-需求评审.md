# 需求评审：风格画布（StyleCanvas）

> 研发/工程视角评审 PRD v0.1，标记实现前必须澄清的问题。🔴 阻塞 / 🟡 需澄清 / 🟢 建议

## 🔴 阻塞项（不解决无法开工）

| # | 问题 | 说明 | 建议 |
|---|------|------|------|
| R1 | "高保真"的验收标准未量化 | 主观，研发无法判断做没做到 | 定义：生成结果须使用风格库的色/字/间距 token，且包含该风格规定的基础组件；以"风格一致性评分 ≥ 90%"为准 |
| R2 | AI 改写的 diff 粒度未定义 | 是整 frame 替换还是字段级 diff？影响实现复杂度 | MVP 先做**区块级 diff**（背景/组件/新增项），不做像素级；明确列出可 diff 的维度 |
| R3 | 风格库的数据结构未定义 | F2 是核心，但"一种风格"包含什么没写清 | 定义 schema：色板 / 字体 / 间距 / 圆角 / 阴影 / 基础组件清单，与 `design-references/` 对齐 |

## 🟡 需澄清

| # | 问题 | 建议 |
|---|------|------|
| Y1 | 生成 ≤ 30s 在复杂界面也成立吗？ | 设超时降级：超时返回"骨架 + 重试"，不卡死 |
| Y2 | 同画布"风格继承"与"单 frame 换风格"冲突时以谁为准 | 以 frame 自身风格为准，画布风格只作新建默认值 |
| Y3 | 回退的范围（单步/多步） | MVP 支持单步撤销栈即可，先不做版本树 |

## 🟢 建议

- G1：生成中占位 frame 给"取消"按钮，避免误生成等待。
- G2：风格库每种风格配一张缩略示意，降低选择成本。
- G3：连线建议支持"从 frame 边缘拖出即连"，减少切工具的动作。

## 评审结论

PRD 方向清晰、差异化成立。**R1–R3 三个阻塞项需在开工前补齐定义**（均与"高保真"和"风格库"这两个核心承诺直接相关）。补齐后可进入交互设计。

回复「继续」进入阶段四：交互设计。
