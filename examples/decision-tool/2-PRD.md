## 一、背景与目标

### 业务背景

现有租房流程高度依赖中介口头推荐和微信聊天记录，缺乏结构化工具辅助决策。真实场景验证（调研日期 2026-04-01）：用户帮朋友在苏州租房，历时周六全天 + 周日半天，实地看房 5+ 套，最终敲定 1 套。

**市场空白**：贝壳等平台利益与用户利益存在错位（帮用户高效说"不"会减少看房次数，影响平台 KPI），导致结构化比较工具在行业内长期缺失。

### 当前痛点

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| 标准渐进激活 | 「小区环境」直到看第一套房才被列入标准 | 白跑无法提前规避 |
| 预筛效率低 | 中介缺乏书面标准，推荐靠经验猜测 | 无效看房占比高 |
| 比较认知负荷高 | 房源信息散落在微信，标准调整后无法重算 | 决策质量低、耗时长 |

### 量化目标

| 指标 | 基线（调研场景） | 目标 |
|------|---------------|------|
| 实地看房套数 | 5+ 套 | ≤ 3 套（视频初筛压缩） |
| 最终决策耗时 | 1.5 天 | ≤ 1 天 |
| 现场录入单套耗时 | — | ≤ 60 秒 |

> 注：基线数据来自单一用户访谈，需上线后扩大样本验证。

## 二、目标用户

| 角色 | 描述 | 使用场景 |
|------|------|---------|
| 主决策者（主要用户） | 自己找房或帮他人找房；需要整理和对比多套房源 | 现场录入、发起对比、管理标准 |
| 协同参与者（间接用户） | 朋友/家人，需要查看对比结果给出意见 | 通过只读分享链接查看 |

**用户特征（调研样本）**：产品经理，有亲身协助他人租房经历，熟悉移动端工具，找房频率约 2-3 年一次。

## 三、用户故事

### 核心故事

**When** 我在看房现场看完一套房，  
**I want to** 用最少操作把这套房的印象记录下来，  
**So that** 我事后还能清楚回想每套房的优缺点，不用翻微信聊天记录。

### 辅助故事

1. **When** 我看完两套房想做比较，  
   **I want to** 看到它们在关键维度上的差异一目了然，  
   **So that** 我能快速淘汰较差的一套，不需要在脑子里同时记住两套房的所有信息。

2. **When** 我意识到某个之前没想到的维度很重要（如小区环境），  
   **I want to** 快速把它加入我的决策标准，  
   **So that** 后续的房源都能用这个标准来评估。

3. **When** 我和朋友共同决策，  
   **I want to** 把当前的对比结果发给朋友看，  
   **So that** 我们不需要用语言描述每套房的情况，直接基于同一份信息讨论。

## 四、功能范围（MVP）

### Must（必须有）

| # | 功能 | 说明 |
|---|------|------|
| M1 | 语音/文字快速录入房源 | 现场输入自由文本，AI 解析为结构化字段 |
| M2 | 贝壳链接解析 | 粘贴链接，AI 自动提取房源信息 |
| M3 | 房源库（卡片列表） | 展示所有候选房源及其状态（候选中/已淘汰） |
| M4 | 锦标赛式对比台 | 选两套房发起对比，高亮差异，淘汰一套 |
| M5 | 标准动态激活 | AI 检测新维度或高频词，提示加入标准 |
| M6 | 标准面板 | 查看和管理已激活标准，支持调整权重级别 |
| M7 | 只读分享链接 | 将当前项目快照上传服务端，生成可分享的只读链接 |

### Should（应该有）

| # | 功能 | 说明 |
|---|------|------|
| S1 | AI 对比摘要 | 对比台底部展示一句话差异总结 |
| S2 | 淘汰原因记录 | 淘汰时可选填一句话原因，供 AI 归纳标准使用 |
| S3 | 标准权重调整后重算 | 调整权重后对比台自动刷新评分 |

### Could（可以有，低优先级）

| # | 功能 | 说明 |
|---|------|------|
| C1 | 分享页拉新入口 | 分享页底部「用同款工具找房」引导按钮 |
| C2 | 项目命名 | 用户可自定义项目名称（默认「{城市}租房 {年月}」） |

### 明确排除（MVP 不做）

- 多人协作编辑
- 买房 / 买车等其他决策场景
- 账号体系与云同步（数据本地存储，分享走快照）
- 与贝壳等平台的深度集成（仅支持链接粘贴解析）

## 五、产品流程

### 主流程

```
[首次打开]
    └─ 创建项目（默认名：{城市}租房 {年月}）
           │
           ▼
[房源库]（主页面）
    ├─ 点击「+ 记录新房源」→ [录入页]
    │       ├─ 输入自由文本或粘贴贝壳链接
    │       ├─ AI 解析 → 结构化字段预览
    │       ├─ 用户可修改字段
    │       └─ 保存 → 返回房源库
    │
    ├─ 点击「发起对比」→ [对比台]
    │       ├─ 选择两套房源
    │       ├─ 展示维度差异（高亮）
    │       ├─ AI 提示新发现维度（可选加入标准）
    │       ├─ 用户点击「淘汰 A」或「淘汰 B」
    │       ├─ 可选填淘汰原因
    │       └─ 返回房源库（被淘汰房源变灰）
    │
    └─ 点击「分享」→ 上传快照 → 生成只读链接

[标准面板]（随时可进入）
    ├─ 查看已激活标准（三级：一票否决 / 重要 / 加分）
    ├─ 调整权重级别
    └─ 处理 AI 建议（接受 / 忽略）
```

### 关键异常路径

```
贝壳链接解析失败
    └─ 提示「链接解析失败，请手动填写」→ 进入手动录入模式

AI 字段解析结果不准确
    └─ 用户可直接点击字段修改 → 修改后保存

分享链接生成失败（网络问题）
    └─ 提示「生成失败，请检查网络后重试」→ 本地数据不丢失
```

## 六、功能详细描述

### M1 语音/文字快速录入

| 要素 | 描述 |
|------|------|
| 触发条件 | 用户点击「+ 记录新房源」，进入录入页 |
| 输入 | 自由文本（手动输入或语音转文字） |
| 处理逻辑 | AI（LLM）从文本中提取：价格、地址、采光、车位、装修、噪音、地铁距离、房东印象等字段；每个字段赋绝对评价状态（好 / 一般 / 差）之一 |
| 输出 | 结构化字段列表（可编辑），用户确认后保存为房源记录 |
| 异常处理 | AI 无法提取某字段 → 该字段显示为空，用户手动填写；语音识别失败 → 降级为文字输入 |

### M2 贝壳链接解析

| 要素 | 描述 |
|------|------|
| 触发条件 | 用户在录入框粘贴内容时，系统自动检测是否为 ke.com 或 anjuke.com 域名链接；若是，录入框下方实时出现「正在解析链接…」提示，**无需用户额外操作**，自动触发 M2 流程 |
| 输入 | 房源详情页 URL |
| 处理逻辑 | 服务端爬取页面内容 → LLM 归纳结构化字段；需处理反爬（UA 伪装、频率限制） |
| 输出 | 同 M1 输出，自动填充字段预览；用户可修改后保存 |
| 异常处理 | 解析失败（反爬/页面结构变更）→ 提示「链接解析失败，请手动填写」，进入 M1 流程 |

### M4 锦标赛式对比台

| 要素 | 描述 |
|------|------|
| 触发条件 | 用户在房源库长按或勾选一张卡片后进入多选模式，再勾选第二张卡片，底部浮现「发起对比」按钮，点击进入对比台 |
| 输入 | 两套房源的结构化字段（含录入时 AI 赋予的绝对状态：好/一般/差）；当前标准列表及权重 |
| 处理逻辑 | **两层展示逻辑**：①每个字段展示录入时 AI 赋予的绝对状态（好/一般/差）；②相对高亮：两套房同一字段对比，绝对状态更优者标 ✅，更差者标 ❌，相同则置灰；若字段状态相同则两侧均不高亮；AI 生成一句话对比摘要 |
| 输出 | 对比矩阵页面；用户点击「淘汰 A」或「淘汰 B」后，被淘汰房源状态更新为「已淘汰」，返回房源库 |
| 异常处理 | 两套房源共同维度不足 3 个 → 提示「这两套房的信息较少，建议补充更多字段后再对比」，但不阻止用户继续 |

### M5 标准动态激活

| 要素 | 描述 |
|------|------|
| 触发条件 | ①对比完成后，检测到本次对比中出现了未在标准列表中的维度；②每次用户**保存新房源时**，检测原始文本中高频词（阈值：同一词/概念在 ≥3 条记录中出现） |
| 输入 | 当前房源记录的原始文本；已激活标准列表 |
| 处理逻辑 | LLM 对比新维度与已有标准，判断是否为新增；统计高频词（阈值：≥3 条记录）；同一维度在当次项目中最多提示 1 次（用户忽略后不再重复） |
| 输出 | 气泡提示「你刚才提到了 XX，要加入标准吗？」→ 一键确认加入（默认级别：重要） |
| 异常处理 | 用户选择「忽略」→ 该维度在当次项目内不再提示 |

### S3 标准权重调整后重算

| 要素 | 描述 |
|------|------|
| 触发条件 | 用户在标准面板调整某维度的权重级别（一票否决 / 重要 / 加分） |
| 输入 | 更新后的标准列表及权重；最近一次对比的两套房源数据 |
| 处理逻辑 | 按新权重重新计算两套房源各维度的相对优劣展示顺序：一票否决维度置顶、重要维度居中、加分维度置底；不做数值加权求和，仅重新排序维度行 |
| 输出 | 对比台维度列表按新权重顺序重新渲染 |
| 异常处理 | 若当前无对比中的房源，调整权重仅保存至标准列表，下次发起对比时生效 |

### M7 只读分享链接

| 要素 | 描述 |
|------|------|
| 触发条件 | 用户点击「分享」按钮 |
| 输入 | 当前项目完整数据（房源列表 + 标准列表） |
| 处理逻辑 | 序列化项目数据 → 上传至服务端 → 生成唯一只读快照 URL；快照不与本地数据同步 |
| 输出 | 可分享的 URL；用户可复制后发送给他人 |
| 异常处理 | 网络失败 → 提示「生成失败，请检查网络后重试」；本地数据不受影响 |

## 七、边界与约束

### 产品边界（不做什么）

- 不替用户做决策，AI 只呈现差异、归纳标准、提供摘要
- 不提供买房 / 买车场景（MVP）
- 不做多人实时协作编辑
- 不强制要求用户登录

### 技术约束

| 约束 | 说明 |
|------|------|
| 数据存储 | 本地优先（IndexedDB），无账号体系 |
| 分享快照 | 需服务端存储，但仅存快照数据，无用户身份绑定 |
| 贝壳链接解析 | 依赖第三方页面结构，存在随时失效风险 |
| 平台 | H5 移动端优先，需适配主流 iOS/Android 浏览器 |
| AI 接入 | 需接入 LLM（如 Claude API），产生 token 成本 |

## 八、验收标准

### 功能验收

| 功能 | 验收标准 |
|------|---------|
| M1 录入 | 用户输入自由文本后，AI 解析成功率 ≥ 90%（至少提取出价格或地址之一） |
| M2 链接解析 | 贝壳链接解析成功率 ≥ 80%（需 PM 确认量化值） |
| M1 录入耗时 | 现场完成一套房录入（含 AI 解析）≤ 60 秒 |
| M4 对比台 | 发起对比到完成淘汰操作 ≤ 3 步 |
| M5 标准激活 | 高频词触发准确率 ≥ 70%（不误报无关词）（需 PM 确认量化值） |
| M7 分享链接 | 生成链接成功率 ≥ 99%（网络正常情况下）；接收方打开只读页面加载时间 ≤ 3 秒 |

### 性能验收

| 指标 | 标准 |
|------|------|
| AI 解析响应时间 | ≤ 5 秒（P90） |
| 对比台渲染时间 | ≤ 1 秒 |
| 本地数据读写 | IndexedDB 操作 ≤ 200ms |
