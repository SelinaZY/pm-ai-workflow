PM AI 工作流

输入一句需求描述，自动完成从探索到验收 Case 的完整 PM 设计闭环

Full 模式 — 阶段零～六全自动串联
Lite 模式 — 仅阶段零 + PRD
0
阶段零 需求探索 pm-brainstorming
💬严格 PM 视角，一问一答探索用户场景、功能形态和方案取舍，禁止讨论技术实现路径
🔍按需调研竞品形态，结论沉淀到设计文档供后续阶段复用，不重复搜索
workspace/brainstorming/YYYY-MM-DD-{feature}-design.md
1
阶段一 需求调研 research
📊JTBD 解构 + PEST 扫描 + 竞品六维分析，输出 GO / NO-GO / 有条件推进 明确决策
🤖竞品资料抓取委派给子 Agent，主会话只接收结构化摘要，不污染上下文
workspace/research/需求调研_{feature}_{date}.md
2
阶段二 PRD 编写 write-prd
📋标准十章节结构，验收标准按「用户场景视角」组织（场景 / 跨场景对比 / 通用规则 / 性能）
🚫技术中立铁律：禁区词扫描确保 PRD 只描述「做什么」，不预设研发实现路径
workspace/prd/PRD_{feature}_v0.1.md
3
阶段三 需求评审 review-prd
🔬独立子 Agent 以研发视角评审，避免主会话「作者偏见」，输出红 / 黄 / 绿三级问题
⚡有阻断问题时自动暂停等待 PM 决策，无阻断则直接串联进入下一阶段
workspace/prd/评审_{feature}_{date}.md
4
阶段四 交互设计 + 交互审查 ixd → review-ixd
📐逐页生成 ASCII 线框图、状态矩阵、跳转规则；改造页面必须逐字段说明处置方式
🔄IxD 产出后自动触发交互审查，审查结论落地后同步更新产品页面知识库（设计态）
workspace/ixd/交互设计说明_{feature}_v0.1.md
5
阶段五 原型设计 prototype
🎨从已采集的线上 DOM 提取组件库，生成风格与产品一致的 HTML 可交互原型
🔵新增元素蓝色虚线标注、改造元素橙色虚线标注，评审者一眼看出变更范围
workspace/prototype/{feature}/*.html
6
阶段六 验收 Case 设计 acceptance · 子流程 A
⚔️攻击性视角推演：每条 AC 至少 2 条 Case（正向 + 边界/异常），覆盖高风险误操作场景
🔗Case 编号与 PRD AC 编号明确关联，可追溯到具体验收标准
workspace/acceptance/{feature}/验收Case_{feature}_v0.1.md
研发实现 / 部署上线
7
阶段七 · 手动触发 验收执行 acceptance · 子流程 B
📸Playwright 自动截图，AI 逐条判定，生成带截图的 HTML 验收报告
✅验收通过 → roadmap 状态更新为「已上线」；不通过则保持「待排期」等待返工
workspace/acceptance/{feature}/验收报告_{feature}_{date}.html

⚡ 自动串联中的阻断条件

调研结论为 NO-GO — 暂停并向 PM 报告，等待决策
需求评审发现阻断问题 — 等待 PM 逐条回复落地后继续
改造页面缺少现状文档（products/pages/）— 强制阻断，不得继续产出
PRD 第八章仍是旧版「功能点平铺」结构 — 要求重构为用户场景视角后再继续
基于 Claude Code · 触发方式：直接描述需求，如「在「你的产品」上增加 X 功能」
