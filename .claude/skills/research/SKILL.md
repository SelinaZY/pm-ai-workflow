# 需求调研与可行性分析

## 交互规则（严格遵守）

### 第一步：确认需求来源与背景

收到需求后，先检查 `workspace/brainstorming/` 下是否有对应的设计文档。

**情况一：有头脑风暴产出**

从设计文档中自动提取「需求来源」「已知背景」「决策记录」，输出确认：

```
从头脑风暴文档中提取到以下信息：

1. **需求来源**：{从文档中提取的来源}
2. **已知背景**：{从文档中提取的背景要点}
3. **已否决方案**（共 {N} 项，调研阶段不再重新评估）：
   - {方案 B} — 否决原因：{...}
   - ...

信息有误或需要补充请回复，没问题回复「继续」。
```

**重要约束**：调研阶段**禁止重新评估已否决方案**。已否决方案在调研报告中以「已否决方案」一节原样保留，仅作上下文说明，不重新做可行性分析。新增竞品对比时，若竞品提供的方案恰是已否决方案，标注「我们已在 brainstorming 阶段否决，原因：{...}」。

**情况二：无头脑风暴产出**

输出以下两个问题，**不要开始分析**：

```
在开始调研前，需要你确认两件事：

1. **需求来源**：这个需求从哪里来？（客户反馈 / 内部提出 / 竞品压力 / 其他）
2. **已知背景**：有没有我应该知道的前提？（如已有内部讨论、技术约束、初步判断等）

没有的话直接回复「没有」。
```

### 第二步：静默执行

收到回答后，**不输出过程**，按下方方法论完整执行 Step 0～Step 4，将结果写入文件：

```
workspace/research/需求调研_{feature}_{YYYY-MM-DD}.md
```

同时将竞品分析结论追加到 `products/{product}/competitive.md`。

### 第三步：输出摘要

文件写完后，只输出以下格式：

```
✅ 阶段一完成：需求调研
📄 workspace/research/需求调研_{feature}_{date}.md

决策：[GO ✅ / NO-GO ❌ / 有条件推进 ⚠️]
优先级：[P0 / P1 / P2 / P3]
关键风险：[1 句话]

关键发现：
- [市场可行性结论 + 1 句理由]
- [技术可行性结论 + 1 句理由]
- [最重要的竞品洞察或差异化结论]

[若有开放问题]
待决策 [N] 项（需确认后才能进入 PRD）：
1. [问题描述] — 影响：[哪个模块]
2. ...

→ 自动进入阶段二：PRD 编写
```

## 方法论（内部执行参考，不输出给用户）

### Step 0：需求来源识别

| 来源类型 | 典型信号 | 可信度 |
|---------|---------|-------|
| 客户 / 用户直接反馈 | 工单、访谈、NPS | 高 |
| 市场信号 | 竞品新功能、行业报告 | 中 |
| 内部驱动 | 战略目标、技术债、运营数据 | 中 |
| 监管 / 合规 | 政策变化、安全要求 | 高 |

### Step 1：需求解构（JTBD + 5 Whys）

JTBD 格式：When [情境]，I want to [行为]，so that [结果]

必须回答四个核心问题：
- 核心问题是什么？（业务痛点，去掉技术描述）
- 目标用户是谁？（行业、角色、规模）
- 现在怎么解决的？
- 成功标准是什么？（尽量可量化）

### Step 2：市场可行性

竞品调研：**通过 Agent 工具委派 subagent 完成**。执行前先检查设计文档是否包含「竞品形态参考」章节：

- **有「竞品形态参考」**：产品形态已在 brainstorming 阶段调研完毕，subagent 只补充「定价 / 市场规模 / 短板 / 对我们的启示」四个商业维度（≤ 400 字）。在报告中注明「产品形态参考来自 brainstorming 阶段竞品调研」并引用原表格。
- **无「竞品形态参考」**：subagent 用 WebSearch + WebFetch 抓取最多 3 个竞品（优先国内）资料，按「定位 / 核心功能 / 技术方式 / 定价 / 短板 / 对我们的启示」六维输出结构化摘要（≤ 600 字）。

主会话基于 subagent 摘要写报告，长输入只占 subagent 上下文，不污染主会话。

PEST 扫描：政策 / 经济 / 社会 / 技术四维快速判断（主会话直接做，无需委派）。

结论格式：**[市场需求强 ✅ / 中等 ⚠️ / 弱 ❌]** + 1-2 句理由

### Step 3：技术可行性

难点分级：低（标准方案）/ 中（需 AI 或复杂工程）/ 高（需定制模型或大工程量）

结论格式：**[技术可行 ✅ / 部分可行 ⚠️ / 风险高 ❌]** + 1-2 句理由

### Step 4：综合决策

决策矩阵：

|  | 技术可行 | 技术存疑 | 技术风险高 |
|--|---------|---------|----------|
| **市场需求强** | ✅ 立即推进 | ⚠️ 先做技术 POC | ⚠️ 评估外包 |
| **市场需求中等** | ⚠️ 评估优先级 | ⚠️ 低优先级 | ❌ 暂不做 |
| **市场需求弱** | ⚠️ 小步验证 | ❌ 不做 | ❌ 不做 |

优先级：P0（当前迭代必须）/ P1（近期 1~2 迭代）/ P2（中期规划）/ P3（有余力再看）

**必须给出明确结论**，信息不足时给"倾向性判断"并列出待验证事项，不能只列待验证事项。

### 报告文件结构

```
需求调研：{feature}
调研日期 / 需求来源
零、已否决方案（来自 brainstorming，不重新评估）  ← 仅当 spec 包含决策记录时
一、需求解构（JTBD + 核心问题 / 目标用户 / 当前解法 / 成功标准）
二、市场可行性（竞品表格 + PEST + 结论）
三、技术可行性（难点分级 + 风险 + 结论）
四、综合决策（RICE / Kano / MoSCoW + GO结论 + 优先级 + 行动项）
五、信息来源（链接列表）
```
