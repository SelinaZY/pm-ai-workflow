# PM 需求探索

通过自然对话把一个模糊想法变成可进入「需求调研」阶段的设计文档。**全程保持 PM 视角**——只关注用户、场景、价值、范围，不讨论实现路径。

<HARD-GATE>
禁止在用户确认设计前调用任何下游 skill（research / write-prd / ixd / writing-plans）或撰写最终设计文档。每一个探索过程都要走完检查清单，不论需求看起来多简单。
</HARD-GATE>

## 反模式：「这个需求太简单不用探索」

小需求（单字段调整、文案优化、单按钮新增）也要走探索过程。简单需求是隐藏假设最多的地方——「我以为大家都默认是 X」往往就是返工源头。设计可以很短（几句话），但 MUST 走完 checklist 并取得用户确认。

## 角色边界（铁律，最高优先级）

PM 探索阶段的任务是**把需求落为用户可感知的功能形态**，不是决定技术实现路径。交互方向、操作路径、功能形态是 PM 要讨论的；技术选型是研发的工作。

**禁讨论清单**（一旦提到立刻回弹）：

| 类别 | 例 |
|------|----|
| 技术栈 | 「用 React/Vue」「用 Spring Boot」「用 Redis」 |
| 算法/模型 | 「用 GPT-4」「用 Embedding」「用动态规划」 |
| 通信/并发 | 「异步 / 同步 / 队列 / 轮询 / 长连接」 |
| 数据结构 | 「加一个 xxx 表 / 字段类型用 varchar / 加索引」 |
| 接口设计 | 「调用 xx 接口 / 返回 JSON / HTTP 状态码」 |
| 架构 | 「微服务 / 单体 / 流式 / 批处理」 |

**用户提到禁讨论项时的回弹模板**：

> 「这是研发 POC 阶段决定的事情，我们这一阶段聚焦在『用户看到什么、怎么操作、完成了什么』。我先把这个想法记下来作为研发的输入，我们继续讨论 [回到上一个 PM 问题]。」

**PM 视角的关注层次**（探索时只在这五层发问）：

1. **用户是谁** — 角色 / 行业 / 规模 / 在产品中的位置
2. **场景与痛点** — 在什么情境下做什么事 / 现在怎么解决 / 痛在哪
3. **期望改变** — 这个需求上线后用户的体验/产出有什么不同
4. **功能形态** — 用户在界面上看到什么、点什么、完成什么（不涉及技术实现）
5. **成功标准** — 怎么衡量做对了（量化指标 / 用户行为 / 业务指标）

## Checklist

你必须为以下每一项创建任务，并按顺序完成：

1. **了解项目上下文** — 检查 `products/<product>/` 文件夹，了解产品定位、现有页面、roadmap
2. **澄清问题，每次一个** — 围绕「用户/场景/期望/功能形态/成功标准」五层，不跨入技术层
3. **识别明确排除项** — 主动追问哪些相关能力**本期不做**，输出「明确排除」清单
4. **竞品调研（按需）** — 若 PM 不确定功能长什么样，用 WebSearch 调研竞品实现，先呈现「竞品 A 是这样做的、竞品 B 是这样做的」，再问 PM 倾向哪个方向，最后将调研结论融入候选方案
5. **提出 2-3 个候选方案** — 在「产品形态/范围/交互方向/用户操作路径」层面展开，不涉及技术实现。给出推荐方案与理由。
6. **呈现设计** — 分章节，每节请求确认
7. **写设计文档** — 保存到 `workspace/brainstorming/YYYY-MM-DD-<topic>-design.md` 并提交
8. **设计文档自审** — 检查占位符、内部矛盾、模糊措辞、技术泄漏
9. **用户审阅设计文档** — 等待用户确认
10. **进入下一阶段** — PM 工作流下进入 `research`（需求调研）

## 探索过程

### 1. 了解项目上下文

- 读 `products/<product>/roadmap.md`、`products/<product>/competitive.md`、`products/<product>/pages/` 下相关页面
- 不读代码仓库或技术文档（PM 探索不需要）
- **对于改造现有产品的新功能，必须额外完成**：
  1. 梳理产品的功能层级结构（如「项目 → 任务 → 详情页」），明确每一层的职责
  2. 在候选方案中显式说明「新功能挂在哪一层、为什么」，不得跳过此讨论直接给出方案
  3. 若方案中出现「新增独立入口」，必须先证明「现有层级无法承载」，否则优先复用现有层级

### 2. 澄清问题

- **每条消息只问一个问题**——不要一次抛多条
- **能用单选题就用单选题**——降低用户负担；开放式问题也可以
- 问题严格围绕「用户/场景/期望/成功标准」四层
- **如果用户的回答里出现技术词，先用回弹模板把它接住，记录到「研发输入」附录，再回到当前问题**

### 3. 识别明确排除

- 在确认了「需求是什么」之后，主动追问「哪些相邻能力**本期不做**」
- 用具体例子提问：「跨页选中要不要做？」「历史记录要不要持久化？」「移动端这次包不包含？」
- 输出独立的「明确排除（Out of Scope）」清单，会原样流入 PRD 第七章和 IxD 约束

### 4. 竞品调研（按需）

**触发条件**：PM 明确表示不确定功能长什么样，或需求描述中缺乏对功能形态的具体设想。

**执行步骤**：

1. 用 WebSearch 搜索 2-3 个竞品对同类功能的实现，关键词围绕「功能名称 + 产品类型 + 截图/介绍」
2. **先呈现竞品现状**：逐个描述「竞品 A 是这样做的：[用户视角的操作路径和界面形态]」，不加评价
3. **再问 PM 倾向**：「你更倾向哪个方向，或者有没有哪个点是你想要/不想要的？」
4. **最后融入候选方案**：把竞品调研结论作为候选方案的参考依据，不单独列为一个方案

**注意**：竞品调研只看「用户看到什么、怎么操作」，不分析竞品的技术实现。

### 5. 提出 2-3 个候选方案

候选方案描述维度（**严格限制在以下 5 项内**）：

- **形态**：是新页面、改造现有页面、还是新流程
- **范围**：覆盖哪些用户场景，先做哪部分，后做哪部分
- **交互方向**：单步/多步、即时/延迟、有无确认环节（不画线框图、不定字段）
- **用户操作路径**：用户点什么、看到什么、完成什么（1-2 步自然语言描述，用户视角）
- **取舍**：每个方案在「价值/复杂度/上线节奏」上的取舍

**禁止**在候选方案里提到：技术实现、表结构、接口、性能数字（除非是用户感知的 SLA）。

给出推荐方案 + 推荐理由。

### 6. 呈现设计

- 按章节呈现，每节后请求确认
- 章节大小匹配复杂度——简单需求几句话即可，复杂需求每节最多 200-300 字
- 必含章节：
  - **背景与痛点**（用户/场景/现状）
  - **核心价值**（这个需求解决了什么）
  - **范围（包含 / 明确排除）**
  - **核心用户旅程**（用户视角的一两条主路径，不是流程图）
  - **成功标准**（如何衡量做对了）
- **不写**：架构图、组件拆分、数据流、错误处理机制、测试策略——这些都是后续阶段的产物

## 写设计文档

- 保存到 `workspace/brainstorming/YYYY-MM-DD-<topic>-design.md`
- 文档结构（与 research/write-prd 衔接）：

```markdown
# {feature 名称} 探索设计

**日期**：YYYY-MM-DD
**产品**：{product}
**触发来源**：{客户反馈/内部提出/竞品/其他}

## 一、背景与痛点
{用户、场景、现状描述}

## 二、核心价值
{这个需求上线后改变了什么}

## 三、范围

### 包含
- ...

### 明确排除（Out of Scope）
- ...

## 四、核心用户旅程
{1-2 条主路径，自然语言描述，不画流程图}

## 五、成功标准
{量化指标 / 用户行为 / 业务指标}

## 六、功能形态描述
{用户视角的关键页面/状态/操作描述，例如：「用户在 X 页面看到 Y，点击后进入 Z 状态，完成 W 操作」。不画线框图，不定字段，只描述用户感知到的形态。}

## 七、竞品形态参考
{仅当第 4 步执行了竞品调研时填写，否则整节删除。格式固定：}

| 竞品 | 用户操作路径 | 界面形态 |
|------|------------|---------|
| {竞品 A} | {用户点什么、看到什么、完成什么，1-2 句} | {页面/弹窗/侧边栏/嵌入式等} |
| {竞品 B} | ... | ... |

**采纳方向**：{说明最终方案参考了哪个竞品的哪个方向，或说明未参考任何竞品}

## 七、研发输入（PM 不决策，仅记录）
{用户在探索过程中提到的技术线索，原样保留供研发 POC 阶段参考}

## 决策记录

### 已采纳方案
- **{决策点 1}**：选择 {方案 A}
  - 理由：{为什么}

### 已否决方案
| 方案 | 否决原因 | 决策日期 |
|------|---------|---------|
| {方案 B} | {原因} | YYYY-MM-DD |

### 暂缓决策（V1 不做，未来再评估）
- **{方案 C}**：理由 {为什么暂缓}
```

## 设计文档自审

写完后用「冷读」视角检查：

1. **占位符扫描**：有没有 TBD/TODO/「待补充」？补上或标注「需 PM 后续确认」。
2. **技术泄漏扫描**：七、研发输入之外的章节有没有出现禁讨论清单中的词？有则改写为用户视角。
3. **范围检查**：「包含」与「明确排除」有没有交叉/矛盾？
4. **歧义检查**：任何一条要求是否可以两种解读？挑一种写明。

内联修复，不再重审。

## 用户审阅闸门

自审过后说：

> "探索文档已写入 `<path>`，请审阅后告诉我是否需要修改，确认后进入需求调研阶段。"

等用户确认。要求修改则改完重审。

## 下一步

- 在 PM 工作流下：进入 `research`（需求调研），把设计文档作为输入
- 独立调用：仅产出探索设计文档，由用户决定下一步

## 关键原则

- **每次只问一个问题**
- **优先单选题** — 降低用户认知负担
- **YAGNI ruthless** — 砍掉不必要的范围
- **永远在 PM 视角** — 出现技术词立刻回弹
- **明确排除 ≥ 明确包含** — 范围管理是 PM 的核心责任
- **决策即记录** — 已采纳/已否决/暂缓三类全留痕，避免下游阶段重新讨论
