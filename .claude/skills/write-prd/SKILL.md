# PRD 编写

## 交互规则（严格遵守）

### 第一步：检测输入来源，判断模式

静默检查以下两个路径：

- `workspace/research/需求调研_{feature}_*.md` → **research 报告**
- `workspace/brainstorming/*{feature}*design.md` → **brainstorming spec**

根据检测结果进入对应模式：

---

**模式一：仅有 research 报告**

- 若调研报告中有**高优先级未确认的开放问题**，将其列出，要求 PM 先确认：

  ```
  PRD 开始前需要你确认 [N] 个高优先级问题：

  1. [问题描述] — 影响：[哪个功能模块]
  2. [问题描述] — 影响：[哪个功能模块]

  请逐条回复后，我开始撰写 PRD。
  ```

- 若无高优先级开放问题，直接说「调研结论清晰，开始撰写 PRD。」然后执行第二步。

---

**模式二：仅有 brainstorming spec**

首先静默读取 spec 中的「决策记录」章节（若存在）。已否决方案**不**作为 PRD 的开放问题或 Could 项；仅在第七章「边界与约束 - 明确排除」中以一行原文标注：「{方案 B}（brainstorming 阶段否决，原因：{...}）」。

输出以下提问，**等待 PM 逐条回复后再执行第二步**：

  ```
  已读取设计文档，开始撰写 PRD 前需要补充 3 个信息：

  1. 需求来源：这个功能从哪里来？（客户反馈 / 内部提出 / 竞品压力 / 其他）
  2. 量化目标：上线后期望改善哪个指标？（如「导出功能使用率」「用户操作步骤减少 N 步」等）
  3. 优先级：当前迭代 Must，还是 Should？
  ```

---

**模式三：两者都有**

优先从 research 报告提取量化数据和市场背景，从 spec 提取功能范围与流程细节。
若 research 报告中有高优先级开放问题，按模式一处理，否则直接执行第二步。

### 第二步：静默执行

**不输出过程**，按下方方法论和当前模式的章节映射完整撰写 PRD，写入文件：

```
workspace/prd/PRD_{feature}_v0.1.md
```

同时在 `products/{product}/roadmap.md` 中追加该 feature 一行（若不存在）。

### 第三步：输出摘要

```
✅ 阶段二完成：PRD 编写
📄 workspace/prd/PRD_{feature}_v0.1.md

功能范围：Must [N] 项 / Should [N] 项 / Could [N] 项
开放问题：共 [N] 个（高优先级 [N] 个，开发启动前需确认）
[高优先级开放问题列表，每条一行]

→ 自动进入阶段三：需求评审
```

## 迭代更新规则

用户反馈修改时：
- **必须用 Edit 工具做局部替换，禁止 Write 整篇重写**（哪怕修改点散落在多章）。整篇重写会让输出 token 翻倍，是 PM 工作流主要成本来源之一
- 仅当出现以下结构性重构时才允许整篇重写：(a) 第八章用户场景视角重构；(b) 整体十章结构调整；(c) 用户明确要求「重写整份 PRD」
- 在文件末尾追加变更记录，不修改历史内容
- **分期一致性检查**：每次迭代更新后，若 PRD 含多期，执行一次全文搜索，确认五～九章中无下期功能的章节、AC、度量行、性能指标混入本期内容
- 关闭开放问题时保留行并标注 `✅ 已确认：[结论]`
- 功能范围变更时同步更新第四、五、六章（用 Edit 逐章替换）
- **流程-功能对齐检查**：每次新增功能模块（M 编号）或功能范围变化后，立即逐一列出第六章所有 M 编号模块，对照第五章产品流程，确认每个模块至少有一条对应的用户路径覆盖。发现缺失立即补充，不等用户提示。触发时机：新增 M 编号模块时、评审落地导致功能范围变化时、任何修改第四章功能范围的操作后

## 方法论（内部执行参考，不输出给用户）

### PRD 标准十章结构

1. **背景与目标**：业务背景 → 当前痛点 → 量化目标（引用调研数据，不重新推导）
2. **目标用户**：主要用户 / 间接用户 / 报告受众（表格）
3. **用户故事**：JTBD 格式，核心故事 + 辅助故事
4. **功能范围（MVP）**：包含（Must）+ 明确排除项
   - **只写本期内容**：第四章只列本期 Must 功能，不写 Should/Could/下期规划，避免研发产生「要不要顺手做」的歧义
   - **后续候选需求**：在第四章末尾加一行引用「后续候选需求见 `plan/{product}/需求池.md`」；下期需求写入需求池，待下次启用时重新经过头脑风暴和调研再决定是否进入 PRD
   - 五～九章同样只写本期内容
5. **产品流程**：ASCII 树形图，含正常路径 + 关键异常路径
6. **功能详细描述**：每个 Must/Should 模块必须有 **触发条件 / 输入 / 处理逻辑 / 输出 / 异常处理**
   - **改造现有页面时额外要求**：必须包含「现有字段处理表」，逐字段说明在新路线下的处理方式（保留 / 修改为 / 隐藏 / 替换为）。不得静默省略现有字段——每个字段必须有明确的处置结论，未说明的字段视为 PRD 遗漏，需补充后才能进入 IxD 阶段
   - **改造现有页面时额外必须包含「现有字段处理表」**：逐字段列出现有页面的每个字段/元素，并明确说明在新路线下的处理方式（保留 / 修改为 / 隐藏 / 替换为）。不得静默省略任何现有字段——省略等同于「研发自行决定」，是返工的根源。
   - 格式示例：
     | 现有字段 | 新路线下处理方式 | 说明 |
     |---------|----------------|------|
     | 目标端数据库 | 替换为「目标端应用技术栈」 | 显示为只读，固定值「Spring Boot + MyBatis」 |
     | 源端数据库 | 保留，但选项收窄 | 仅 Oracle 可选，其他置灰 |
     | 项目名称 | 保留 | 无变化 |
7. **边界与约束**：产品边界（不做什么）+ 技术约束
8. **验收标准**：按「用户实际使用场景」组织，分四节：
   - **8.1 用户场景验收**：枚举 N 类用户使用场景（每类含「用户动机」「操作路径」），每个场景下挂载该路径必须验证的 AC，编号格式 `AC-{场景字母}{序号}`（如 AC-A1）
   - **8.2 跨场景对比验收**：识别「UI 看着等价、产物完全不同」的危险组合，编号 `AC-X{n}`
   - **8.3 通用规则验收**：与具体使用场景无关的全局规则（权限、数据完整性等），编号 `AC-G{n}`
   - **8.4 性能验收**：编号 `PF-{n}`
   - **必须可量化**（不写"功能正常"，写"成功率 ≥ 95%"）
9. **度量指标**：上线后追踪的指标与目标值
10. **开放问题**：问题 / 影响范围 / 优先级（高优先级 = 开发启动前必须确认）

### 文件头格式

```markdown
# PRD：{feature}

**版本**：v0.1（初稿）
**状态**：待评审
**创建日期**：{YYYY-MM-DD}
**关联调研**：workspace/research/需求调研_{feature}_{date}.md
```

### 章节来源映射（按模式）

| PRD 章节 | 模式一（research） | 模式二（spec） | 模式三（两者） |
|---------|-----------------|--------------|--------------|
| 背景与目标 | research 市场数据 + 痛点 + 量化目标 | spec 背景 + PM 补充的量化目标 | research 量化数据优先 |
| 目标用户 | research JTBD 解构 | spec 场景描述中提取，标注「待调研验证」 | research 优先 |
| 用户故事 | research JTBD 格式 | 从 spec 流程推导 | 合并 |
| 功能范围 | research GO 结论推导 | spec 功能范围直接映射 | spec 优先 |
| 产品流程 | research 推导 | spec 核心流程直接映射 | spec 优先 |
| 功能详细描述 | research 推导 | spec 交互细节展开为五要素 | spec 优先 |
| 边界与约束 | research 技术约束 | spec 明确排除项 + 异常处理 | 合并 |
| 验收标准 | research 成功标准量化 | 从 spec 流程推导，**标注「需 PM 确认量化值」** | research 量化值优先 |
| 开放问题 | research 开放问题 | spec 未覆盖章节自动列为待确认项 | 合并去重 |

### 技术中立原则（铁律，最高优先级）

PRD 描述**「做什么」与「用户能感知到的产物/行为」**，**不**描述「怎么做」。技术选型属于研发职责，PM 在 PRD 里预先决定会越权并锁死研发空间。

**禁区词清单**（出现即触发自检）：

| 类别 | 禁用词示例 |
|------|-----------|
| 通信/并发 | 异步、同步、队列、消息、MQ、Kafka、轮询、长连接、WebSocket、线程、协程、并发、并行、串行、流水线、阻塞、回调、重试 N 次 |
| 存储/缓存 | Redis、缓存、索引、分库、分表、表结构、字段类型、连接池、缓冲区 |
| 算法/模型 | 具体算法名（如 BFS/动态规划）、具体模型名（如 GPT-4/BERT）、Embedding、向量检索、相似度算法 |
| 架构/框架 | 微服务、单体、Spring Boot（除非是产物本身）、MyBatis（除非是产物本身）、流式、批处理、分布式、具体前端组件名（如 Monaco Editor） |
| 接口契约 | 接口字段标识符（如 `migration_type=db_to_db`）、HTTP 状态码、前后端职责分配（「由后端...前端不做」） |
| 性能边界泄漏 | 「P95 从队列到落库」「Job 调度延迟」等暴露内部组件的描述 |

**例外判定准则（核心）**：

- ✅ **合法产物**：用户在屏幕上 / 导出文件里**能直接看见的字符串** —— 如 `@Service` 注解、固定 package `com.example.generated.xxx`、文件名格式 `{sp_name}_v{n}.zip`、中文术语「迁移类型」「迁移到目标数据库」
- ❌ **越权实现**：用户**看不见的执行机制** —— 队列、缓存、重试策略、并发模式、接口字段标识符（`migration_type=db_to_db`）、前后端分工

判定方法：闭眼想用户操作完成后能截图给你看的内容——能截图的属产物（合法），看不见的属实现（越权）。

例外仅限「功能详细描述」的产物字段，**不得**扩散到其他章节（背景、流程、验收、度量）。

**改写规则**：

| 越权写法 | 中立写法 |
|---------|---------|
| 任务进入队列后 5 秒内返回 | 用户提交后 5 秒内看到处理结果 |
| 异步处理避免阻塞 | 提交后用户可继续操作其他功能 |
| 端到端 P95（从队列到落库） | 端到端 P95（从用户提交到结果可见） |
| 用 Redis 缓存最近查询 | 重复查询相同条件时响应时间 ≤ N 秒 |
| 用 XX 算法做相似度匹配 | 系统识别出与目标差异度 ≤ N% 的项 |
| 单层超时自动重试一次 | 失败时呈现为任务异常，用户可从 X 入口重新触发 |
| 严格串行执行三层 | 任一层失败则任务整体失败、不展示半成品 |
| 由后端基于 migration_type 决定，前端不做推断 | 不同迁移类型展示对应的代码语法高亮 |
| 接口返回 `editor_language: "java"` | 代码以 Java 语法高亮显示 |

**易混淆边界（已校准）**：
- 「2 秒内响应」= 用户感知 SLA ✅ | 「超时重试一次」= 研发策略 ❌
- 中文术语（迁移类型/迁移到目标数据库）✅ | 字段标识符（`migration_type=db_to_db`）❌（仅可入「数据契约」附录）
- `@Service` 注解 ✅（用户在 ZIP 看得见） | Monaco Editor ❌（具体前端组件名）

**自检步骤**（写完每章后执行）：
1. 通读本章，圈出禁区词清单中出现的词
2. 判断是否属于「产物本身」例外（能否截图给用户看？）
3. 否则改写为用户视角的可观察行为或产物描述
4. 性能/度量章节特别检查：测量边界是否暴露了内部组件名

### AI 行为约束

- 优先从上游文档提取结论，不重新推导
- 背景章节必须引用来源数据（research 有则引用，无则标注「待补充」）
- 每个功能必须有五要素（触发 / 输入 / 处理 / 输出 / 异常）
- 验收标准第八章必须按「用户场景视角」组织（8.1~8.4 四节结构），**禁止**回到「按功能点平铺」的旧写法
- 用户场景必须显式列出「用户动机」和「操作路径」，AC 挂载到对应场景下
- 必须显式包含「跨场景对比」节（8.2），即使只有 1~2 条也要列出，用于发现易混淆的产物差异
- 验收标准必须可量化；模式二下无量化数据时标注「需 PM 确认量化值」
- 开放问题不能为空；模式二下至少列出验收标准未量化的条目
