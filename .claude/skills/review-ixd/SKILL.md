# IxD 交互审查

## 触发方式

- **自动触发**：ixd Skill 产出文档后自动执行
- **手动触发**：用户说「审查交互」「review ixd」

## 执行步骤

### Step 1：用独立子 Agent 执行审查

**重要**：审查**必须**通过 `Agent` 工具在独立子会话中执行，避免主会话的「作者偏见」（主会话刚生成 IxD，潜意识会偏向「我画的设计没问题」）。

调用方式：

```
Agent(
  subagent_type: "general-purpose",
  description: "审查 IxD",
  prompt: """
你是一位独立的交互设计审查员，没有这份 IxD 文档的编写历史上下文，请完全基于文件内容审查。

待审查 IxD 文档：workspace/ixd/{最新版本，或用户指定路径}
审查知识库：.claude/skills/review-ixd/ixd-review-checklist.md

请严格按 `.claude/skills/review-ixd/SKILL.md` 的「方法论」章节（Step A 逐条审查 + Step B 反向更新预览）执行：

A. 逐条审查
- 对 IxD 文档中的每个页面，逐一应用知识库中的每条审查条目
- 对每个用户操作，检查「操作必要性」类条目
- 对每项信息展示，检查「信息必要性」类条目
- 对每条用户路径，检查「路径最短性」类条目
- 判定规则：只有符合条目「判定标准」的才标记为问题，不过度解读
- 必须给出具体可执行的建议方案，不能只提问题

B. 改造页面现状一致性核查
- 从 IxD 文档中提取所有「改造现有页面」的变更点
- 对每个改造页面，**必须读取** `products/<产品名>/pages/<页面名>.md` 完整文档（若存在）
- 逐项核查：IxD 文档中的改造描述是否与 pages 文档中的现状结构一致？有无遗漏现有元素、有无与现状矛盾的设计？
- 若 IxD 改造章节缺少「现有结构」描述，标记为审查问题（R-xx），要求补充
- 输出变更预览（不要直接写入），格式见 SKILL.md「变更预览格式」

返回时只需告诉我：
1. 审查发现的全部问题清单（R-01 起，按 SKILL.md 格式四要素：页面/触发条目/问题描述/建议方案）
2. 页面知识库变更预览清单（按 SKILL.md 格式：页面名 + 新增/修改/删除项）
3. 若审查无问题：明确说明「未发现待优化项」

**不要**自行写入 IxD 文档或 pages 文档——这些动作由主会话在 PM 决策后执行。
"""
)
```

子 Agent 完成后，主会话仅做汇总展示与决策落地（下方 Step 2~4），**不**重复执行审查逻辑。

### Step 2：输出审查报告

子 Agent 返回后，主会话按以下格式向 PM 输出：

```
🔍 IxD 交互审查完成，发现 [N] 个待优化项：

R-01 [问题标题]
  - 页面：[页面编号和名称]
  - 触发条目：[C-xx 条目名称]
  - 问题描述：[当前设计是什么，为什么有问题]
  - 建议方案：[具体的交互优化方案，足够详细可直接落地]

R-02 ...

请逐条回复「采纳」「不采纳」或给出你的方案。
```

若子 Agent 返回未发现问题：

```
🔍 IxD 交互审查完成，未发现待优化项。
```

### Step 3：执行用户决策

- 用户回复「采纳」的条目：直接修改 IxD 文档
- 用户回复自定义方案的条目：按用户方案修改 IxD 文档
- 用户回复「不采纳」的条目：不做修改

修改完成后重新读取修改区域，确认无误。

### Step 4：反向更新产品页面知识库（设计态）

IxD 审查完成且用户决策全部落地后，把子 Agent 已经准备好的变更预览展示给用户确认：

```
📝 页面知识库更新预览（设计态）

页面：{页面名}.md
变更项：
  + 新增：{描述，如「操作列新增『性能优化』按钮」}
  ~ 修改：{描述，如「工具栏『导出目标端SQL』改为『批量导出』」}
  - 删除：{描述}

是否更新？回复「确认」写入，「跳过」不更新。
```

用户确认后，将变更合并到 pages 文档，并更新文档头部元信息：
- `> 状态：设计态`
- `> 关联需求：{IxD 文档路径}`
- `> 最后更新：{YYYY-MM-DD}`

若对应页面的 pages 文档不存在，提示用户：该页面尚无现状文档，建议先用 `/capture-page` 采集后再更新。

## 方法论（供子 Agent 参考的审查规则）

### Step A：逐条审查

对 IxD 文档中的**每个页面**，逐一应用知识库中的每条审查条目：

- 对每个页面的每个用户操作（按钮、确认、选择），检查所有「操作必要性」类条目
- 对每个页面的每项信息展示（代码块、状态标签、说明文字），检查所有「信息必要性」类条目
- 对每条用户路径（从入口到完成任务的完整链路），检查所有「路径最短性」类条目
- 未来新增的维度类别同理，逐条应用

**判定规则**：只有符合条目中「判定标准」的才标记为问题，不过度解读。

### Step B：变更预览格式

```
📝 页面知识库更新预览（设计态）

页面：{页面名}.md
变更项：
  + 新增：...
  ~ 修改：...
  - 删除：...
```

## 注意事项

- 审查范围仅限交互流程层面，不审查视觉设计、文案措辞等
- 不重复 IxD Skill 已有的设计红线检查
- 发现问题时必须给出具体可执行的建议方案，不能只提问题不给方案
