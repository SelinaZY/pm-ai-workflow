# PRD 研发评审

## 交互规则（严格遵守）

### 第一步：用独立子 Agent 执行评审

**重要**：评审**必须**通过 `Agent` 工具在独立子会话中执行，避免主会话的「作者偏见」（主会话刚写完 PRD，潜意识会偏向「我写的东西没问题」）。

调用方式：

```
Agent(
  subagent_type: "general-purpose",
  description: "评审 PRD",
  prompt: """
你是一位资深研发工程师，独立评审一份 PRD。你**没有**这份 PRD 的编写历史上下文，请完全基于文件内容评审。

PRD 文件路径：workspace/prd/PRD_{feature}_v{version}.md

请严格按照 `.claude/skills/review-prd/SKILL.md` 的「方法论」章节（Step 1~6）执行评审，包括：
- Step 1：内部一致性
- Step 2：技术可行性
- Step 3：完整性
- Step 4：边界与异常覆盖
- Step 5：技术中立检查（PM 越权防线）
- Step 6：验收标准质量（含 6.1~6.4）

评审报告写入：workspace/prd/评审_{feature}_{YYYY-MM-DD}.md，使用 SKILL.md 中的「评审报告文件格式」。

返回时只需告诉我：(a) 评审报告文件路径、(b) 阻断/澄清/建议三类计数、(c) 全部 🔴 阻断问题的标题、位置、问题、建议四要素列表。不要在返回内容中重复整份报告。
"""
)
```

子 Agent 完成后，主会话仅做汇总展示（下方第二步），**不**重复执行评审逻辑。

### 第二步：输出评审结论，等待 PM 逐条回复

```
✅ 阶段三完成：需求评审
📄 workspace/prd/评审_{feature}_{date}.md

评审汇总：🔴 阻断 [N] 个 / 🟡 需澄清 [N] 个 / 🟢 建议 [N] 个

[若有 🔴 阻断问题，在这里逐条列出，格式如下]
🔴 1. [问题标题]
   位置：[章节]
   问题：[具体描述]
   建议：[修改方向]

🔴 阻断问题存在时：暂停流程，等待 PM 逐条回复，落地后自动继续。
若无阻断问题：自动进入阶段四：交互设计。
```

### 第三步：落地 PM 回复

PM 回复后：
- 静默将所有确认结论更新到 PRD 对应章节
- 在 PRD 末尾追加变更记录
- 需要技术 POC 验证的标注 `⚠️ 待技术验证`
- 输出：

```
PRD 已更新，[N] 条评审意见已落地。
[仍有未解决问题时列出]

→ 自动进入阶段四：交互设计
```

## 方法论（内部执行参考，不输出给用户）

### 角色定位

以**研发工程师**视角审读——不质疑业务价值，只关注"能不能按文档实现、有哪里描述不清"。

### Step 1：内部一致性

- 功能范围（第四章）与功能详细描述（第六章）是否对应
- 产品流程（第五章）与功能详细描述的输入/输出是否一致
- 边界约束（第七章）与流程/功能是否矛盾
- 已关闭的开放问题，对应章节是否同步
- **排除项 × AC 交叉对照**：第四章「明确排除」的每一项，到第八章找是否存在验收场景恰好踩在排除边界上（例：排除「跨页选中」vs 场景「全选当前页导出」）。这类矛盾最常被漏，是高频阻断源
- **同概念多处口径一致**：同一规则（如「跳过判断」「权限校验」「同名命名」）在第五章流程、第六章详细描述、第八章 AC 中所有出现位置逐字对比，微差即问题
- **流程图出口完整性（双向核查）**：
  - 正向：第五章流程图中的每个操作节点（按钮点击、状态变更、页面跳转、文件下载）必须能在第六章找到对应的功能描述；流程图中出现的每条箭头都要有功能描述支撑
  - 反向：第六章每个功能的触发条件和输出产物，必须能在第五章流程图中找到对应的入口和出口路径；功能描述中存在但流程图中缺失的节点视为遗漏
  - 典型漏检案例：功能描述新增了「导出」按钮，但流程图未更新对应出口；功能描述说明了状态等待逻辑（如「处理中禁用查看」），但流程图直接跳过等待节点

### Step 2：技术可行性

- 数据库/中间件限制：描述的操作是否符合目标技术栈能力
- 数据流向是否可行
- 性能假设是否合理（"实时返回"是否实际可行）
- 跨系统对比时数据格式是否可映射

### Step 3：完整性

逐一检查每个功能模块是否明确了：量化阈值、数据生命周期、上传格式、失败处理、前置依赖、并发隔离。

### Step 4：边界与异常覆盖

对产品流程每个 Step 追问：
1. 这一步失败会怎样？
2. 用户输入格式错误如何提示？
3. 上游数据缺失如何处理？
4. 有没有边界情况被假设为正常处理？
5. **异常分支是否闭环**：不仅描述现象（「token 失效」「job_id 丢失」），还要给出「用户能做什么 + 系统停在什么状态」。只写现象不算闭环。

### Step 5：技术中立检查（PM 越权防线）

PRD 应描述「做什么 + 用户感知的产物/行为」，禁止预先决定研发实现路径。逐章扫描禁区词：

**禁区词清单**：
- 通信/并发：异步、同步、队列、消息、MQ、Kafka、轮询、长连接、WebSocket、线程、协程、并发、并行、串行、流水线、阻塞、回调、重试 N 次
- 存储/缓存：Redis、缓存、索引、分库、分表、表结构、字段类型、连接池
- 算法/模型：具体算法名、具体模型名、Embedding、向量检索、相似度算法
- 架构/框架：微服务、单体、流式、批处理、分布式、具体前端组件名（如 Monaco Editor）
- 接口契约：接口字段标识符（如 `migration_type=db_to_db`）、HTTP 状态码、前后端职责分配（「由后端...前端不做」）
- 性能边界泄漏：「P95 从队列到落库」「Job 调度延迟」等暴露内部组件

**例外判断（核心准则）**：

- ✅ 合法产物：用户在屏幕上 / 导出文件里**能直接看见的字符串**（注解、固定 package、文件名格式）
- ❌ 越权实现：用户**看不见的执行机制**（队列、缓存、重试策略、并发模式、接口字段标识符、前后端分工）

判定方法：能截图给用户看的属产物（合法），看不见的属实现（越权）。

例外仅限「功能详细描述」的产物字段；其他章节（背景、流程、验收、度量、开放问题）一律视为越权。

**易混淆边界（已校准）**：
- 「2 秒内响应」= 用户感知 SLA，✅ 合法
- 「超时重试一次」= 研发策略，❌ 越权 → 改为「失败时呈现为异常」
- 「面向用户中文术语：迁移类型 / 迁移到目标数据库」= ✅ 合法
- 「字段标识符 migration_type 取值 db_to_db」= 数据契约，❌ 越权（仅可放数据契约附录或删除）
- 「Service 类含 @Service 注解」= 用户在 ZIP 中看得见，✅ 合法（产物本身例外）
- 「Monaco Editor 语言模式 = java」= 具体前端组件名，❌ 越权 → 改为「Java 语法高亮」

**判定规则**：
- 出现禁区词且无例外 → 🟡 需澄清（标注「越权研发选型」），建议改写为用户视角
- 性能度量的测量边界引用了内部组件名 → 🔴 阻断（这会锁死研发实现）
- 同一规则在 PRD 和 IxD 都出现技术细节 → 🔴 阻断

### Step 6：验收标准质量

**6.1 结构性检查（用户场景视角）**

- [ ] 第八章是否按「用户实际使用场景」组织？（8.1 用户场景 / 8.2 跨场景对比 / 8.3 通用规则 / 8.4 性能）
- [ ] 每个用户场景是否显式列出「用户动机」「操作路径」？
- [ ] AC 编号是否采用 `AC-{场景字母}{序号}` / `AC-X{n}` / `AC-G{n}` / `PF-{n}` 格式？
- [ ] 是否识别了"UI 看着等价、产物完全不同"的危险组合并写入 8.2 跨场景对比？（例：全选当前页 vs 不勾选导出全部）
- [ ] 是否有用户会误判的操作（"全选 = 全部"类）？是否有提示文案兜底？

若第八章仍是「功能点平铺」的旧写法（无场景分层、无跨场景对比），**必须作为高优先级问题提出**，要求 PM 重构。

**6.2 内容量化检查**

| 要求 | 不合格 | 合格 |
|------|--------|------|
| 有明确数值 | "功能正常" | "成功率 ≥ 95%" |
| 测量方式可操作 | "结果准确" | "与手工测试误差 < 10%" |
| 测量对象明确 | "性能达标" | "上传到输出 < 5 分钟" |

**6.3 AC 可验证性过滤（非数值类）**

每条 AC 自问：测试能否写出 pass/fail 明确的用例？以下是红旗词，必须转化为可观察行为：

| 红旗类型 | 反例 | 正例（可观察行为） |
|---------|------|------------------|
| 主观动词 | 「让用户停下来核对」「让用户感知」 | 「弹窗统计行显示『已选 N 个任务』」 |
| 否定式模糊 | 「不允许同步阻塞」「不应该卡顿」 | 「弹窗关闭 + 页面顶部出现进度条 + 其他元素可点击」 |
| 未量化程度 | 「快速」「及时」「友好提示」 | 「≤ 5 秒」「错误码 + 文案『...』」 |

**6.4 产物名词收窄检查**

当需求存在 OR 选项（如「ZIP 或合并单文件」「PDF 或 Excel」），检查 AC 里的产物名词是否被默认收窄为其中之一：

- 反例：「ZIP 仅包含全部转换成功的任务」（用户选合并单文件时此 AC 失效）
- 正例：「**导出产物**仅包含全部转换成功的任务」

规则：除非 AC 明确针对某一具体格式，否则用「产物」「文件」等上位词。
