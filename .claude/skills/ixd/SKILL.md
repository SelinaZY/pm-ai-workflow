# 交互设计说明生成

## 交互规则（严格遵守）

### 第一步：静默执行

静默读取对应 PRD（自动查找 `workspace/prd/` 下最新版本），按下方方法论完整生成交互设计说明，写入：

```
workspace/ixd/交互设计说明_{feature}_v0.1.md
```

### 第二步：输出摘要 + 待确认项

```
✅ 阶段四完成：交互设计说明
📄 workspace/ixd/交互设计说明_{feature}_v0.1.md

涉及页面：[N] 个（改造现有 [N] 个 / 新增 [N] 个）

[若有待确认项]
设计待确认 [N] 项（PRD 未定义、设计必须选择方案）：
D-01 [问题描述] — 影响：[页面] — 可选方案：A / B
D-02 ...

待确认项不阻断流程，自动继续。用户可在最终交付后统一回复调整。
```

### 第三步：自动触发交互审查

文档产出后，**立即**执行 `/review-ixd` Skill，对刚产出的 IxD 文档进行交互审查。审查结果追加在摘要之后输出，供用户一并决策。

若涉及页面数 > 0，且所有改造页面的现状文档均已确认存在（Step 2.5 无阻断），**立即**执行 `prototype` Skill，无需等待用户确认。

若涉及页面数 = 0（纯后端逻辑），追加：

```
本需求不涉及页面变更，跳过原型设计。
→ 自动完成工作流。
```

## 方法论（内部执行参考，不输出给用户）

### 角色定位

以**交互设计师**视角拆解 PRD——不重新定义功能范围，只关注"需要几个页面、每个页面放什么、怎么跳转、边界状态是什么"。

### Step 1：提取设计输入

从 PRD 提取：产品流程步骤、各步骤输入/输出、异常路径、技术约束、数量/大小限制、功能范围排除项、验收标准。

### Step 2：规划页面清单

分类：改造现有页面 / 新增向导页 / 新增结果页 / 新增管理页

输出表格：编号 / 页面名称 / 类型 / 对应 PRD 步骤

### Step 2.5：检查改造页面的现状知识库

对 Step 2 中标记为「改造现有页面」的每个页面：
1. 检查 `products/<产品>/pages/<page>.md` 是否存在
2. 若缺失，**立即停止执行**，输出阻断提示并等待用户补充，不得继续进入后续步骤：
   「🚫 阻断：以下改造页面缺少现状文档：[页面名]。请补充后再继续（可发 Figma 链接或手动描述页面现状）。」
3. 若已有文档，**直接读取完整 pages 文档**，不做摘要压缩。改造设计必须基于完整现状，不得遗漏现有元素结构。改造页面的 IxD 章节必须明确写出：「现有结构」（引用 pages 文档中的相关章节原文）+ 「本次变更」（新增/修改/删除了什么），让读者无需对照原文就能理解改动范围。
4. **逐字段处置检查**：对现有页面的每一个字段/元素，必须在「本次变更」中显式说明新路线下的处置方式（保留 / 修改为 / 隐藏 / 替换为）。若 PRD 未定义某字段的处置方式，**不得自行决策**，必须将其列为设计待确认项（D-xx），等待 PM 决策后再设计。

### Step 3：逐页设计

每个页面必须包含：

- **页面功能**：一句话说明核心职责
- **ASCII 线框图**：体现标题区 / 主操作区 / 内容区 / 底部操作区
- **元素说明**：按钮禁用条件、输入字段规则、列表列定义、状态标签、默认值来源
- **状态矩阵**：正常 / 空状态 / 加载中 / 局部失败 / 全部失败
- **出口（跳转规则）**：操作 → 跳转目标 → 条件

**改造现有页面时额外必须包含「现有字段处理表」**：逐字段列出现有页面的每个字段/元素，明确说明在新路线下的处理方式（保留 / 修改为 / 隐藏 / 替换为）。不得静默省略任何现有字段——省略等同于「设计师自行决定」，是与 PRD 产生偏差的根源。格式示例：

| 现有字段 | 新路线下处理方式 | 说明 |
|---------|----------------|------|
| 目标端数据库 | 替换为「目标端应用技术栈」 | 显示为只读，固定值「Spring Boot + MyBatis」 |
| 源端数据库 | 保留，但选项收窄 | 仅 Oracle 可选，其他置灰 |
| 项目名称 | 保留 | 无变化 |

### Step 4：整理待确认项

只提 **PRD 未定义但设计必须选择方案** 的决策点，不重复 PRD 已有内容。

### 设计红线（违反则必须修改）

| 红线 | 说明 |
|------|------|
| 不允许"先建系统再用系统" | 用户第一个操作必须直接产生价值 |
| 高动机页面不超过 2 个决策点 | 高压现场只做一件事 |
| 所有输入字段必须标注默认值来源 | 能预填的必须预填 |
| 主 CTA 文案描述用户行为 | 用"记录这套房"而非"创建条目" |

### 文件头格式

```markdown
# 交互设计说明：{feature}

**版本**：v0.1
**状态**：待设计师评审
**关联 PRD**：workspace/prd/PRD_{feature}_{version}.md
**创建日期**：{YYYY-MM-DD}
```

### 文件尾必须包含变更记录

```markdown
## 变更记录

| 日期 | 版本 | 变更内容 |
|------|------|---------|
| {YYYY-MM-DD} | v0.1 | 初稿创建 |
```

每次审查落地、PM 反馈修改后，追加一行变更记录，不修改历史内容。

ARGUMENTS: $ARGUMENTS
