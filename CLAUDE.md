# Claude Code 协作规则

## 通用规则

- **每次回复以 "Vincy," 开头**：每条回复的第一个词必须是 "Vincy,"，用于监控上下文是否过载。
- **不确定时先询问**：遇到需求设计或代码实现有任何不确定的地方，必须先向 Vincy 提问，**不得**自行假设后直接行动。
- **主语言为中文**：PRD、调研报告、交互设计说明、评审报告等所有 PM 产出文件均用中文编写，除非明确指定其他语言。
- **优先使用已有 Skills**：执行任何 PM 任务前，先检查 `.claude/skills/` 目录中是否有对应 Skill 文件。**不得**创建新 Skill 或抓取外部内容，除非用户明确要求。
- **不主动扩展范围**：只做用户要求的事。不添加功能、不做额外重构、不生成未要求的文档。

## 环境

- **文件编码一律 UTF-8**：所有 Python 脚本的文件 I/O 必须显式指定 `encoding="utf-8"`，禁止依赖系统默认编码（Windows 默认 GBK 会导致乱码）。

## 设计与产品工作

- **改造页面前先核查现状**：涉及已有页面的改造时，必须先读 `products/<产品>/pages/` 下的对应文件，了解当前页面结构（Tab、入口、字段等），**不得**假设后直接提案。文件缺失时，阻断流程并提示用户补充。
- **原型设计参考三级兜底**：生成原型时按「① 线上 DOM → ② 产品声明的设计参考（`products/<产品>.md` 的 `设计参考:` 字段 / `design-system.md` / `.claude/skills/prototype/design-references/`）→ ③ Ant Design」择优。**禁止**给文档、画布、C 端等非后台型产品直接套 Ant Design 兜底——遇到无 DOM 的全新产品，先确认其品类并选/建对应设计参考，再生成。
- **IxD 问题自动归纳**：当用户在 IxD 交付后指出交互问题或要求修改时，**主动**将问题归纳为新的审查条目，提议追加到 `.claude/skills/review-ixd/ixd-review-checklist.md`，经用户确认后写入。
- **MCP 排错优先级**：遇到 MCP 连接问题，第一步检查 `.mcp.json` 中的版本路径是否正确，**不得**进入扩展内部或尝试多个推测性修复。路径不确定时，直接问用户。

## 文档编辑规则

- **手术式编辑**：编辑文档时只修改目标内容，**严禁**覆盖或替换相邻章节。
- **编辑后验证**：每次编辑后重新读取修改区域周围的段落，确认没有意外删除内容。
- **变更传播**：修改任何 PM 产物（调研报告、PRD、评审报告、IxD）中的需求细节时，必须检查关联的上下游产物是否引用了同一内容并同步更新。关联关系：
  - 调研报告 → PRD（背景、JTBD、成功标准）
  - PRD → 评审报告（引用的章节内容）
  - PRD → IxD（功能范围、字段定义、流程步骤）
  - IxD → pages/ 知识库（设计态字段、枚举值）
- **Skill 流程变更传播**：修改 pm-workflow.md 或任何阶段 skill 的流程规则时，必须查阅 pm-workflow.md 中的「Skill 文件依赖关系」矩阵，逐一检查关联文件。

## 错误处理

- **API 错误时立即存档**：遇到 API 配额、鉴权或连接错误时，立刻将已完成内容保存到 `-checkpoint.md` 文件，并告知用户确切中断位置，以便用「继续」恢复。

## 工作流说明

本仓库是一套基于 Claude Code 的 PM AI 工作流，八个阶段（零~七），支持 Lite / Full 双模式：

```
需求探索（Brainstorming） → 需求调研 → PRD 编写 → 需求评审 → 交互设计 → 原型设计（可选） → 验收 Case 设计 → 验收执行（手动）
```

- **Full 模式**：Brainstorming 确认后，阶段一~六自动串联执行，一次性交付全部产出。阶段七在研发上线后手动触发。
- **Lite 模式**：小需求（文案/字段/单页改造/用户声明轻量）仅跑阶段零 + 阶段二（PRD），跳过其余阶段。

Brainstorming 阶段与用户交互确认方案，遇到阻断性问题自动暂停报告。详见 `README.md`。
